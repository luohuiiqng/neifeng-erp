# 生产订单流程说明（执行版）

## 1. 业务目标
- 由订单部统一建单并下发，车间主任负责判断是否需要设计参与。
- 采购为按需分支：物料不足才采购，物料充足可直接备货/生产。
- 备货与生产必须分开展示，不在同一块列表里混合。
- 出货由厂长执行，是否允许出货由订单部根据回款情况决定。

## 2. 主状态流转
`草稿 -> 已下发 -> 设计中(可选) -> 备货中 -> 生产中 -> 待出货 -> 已结案`

说明：
- 若不需要设计：`已下发 -> 备货中`
- 若需要设计：`已下发 -> 设计中 -> 备货中`
- 采购不单独作为主状态，仅作为备货阶段的业务动作。

## 3. 各状态操作人（谁可推进）
| 状态 | 谁可推进到下一状态 | 推进条件 |
|---|---|---|
| 草稿 | 订单部 | 点击订单下发 |
| 已下发 | 车间主任 | 判读是否需要设计 |
| 设计中 | 车间主任（设计完成后） | 设计确认完成 |
| 备货中 | 车间主任 | 备货完成（可并行生产准备） |
| 生产中 | 车间主任 | 生产完成，转待出货 |
| 待出货 | 厂长（执行出货） + 订单部（放行） | 订单部确认回款条件允许 |
| 已结案 | 财务提交/订单部审批（按系统审批流程） | 回款与流程完结 |

## 4. 可见性与编辑规则
- 草稿：仅订单部、管理员可见与可编辑。
- 已下发/设计中/备货中/生产中/待出货：相关业务部门可见。
- 状态推进按钮只对对应角色显示（避免误操作）。
- 已结案默认只读（除管理员外不再改状态）。

## 5. 采购规则
- 备货阶段发现缺料：走采购流程。
- 备货阶段物料充足：无需采购，直接推进。
- 采购执行不改变主状态链路，只影响备货完成时点。

## 6. 展示规则
- 工作台和列表中“备货中”与“生产中”分开展示。
- 不允许用一个“在制”状态混合替代两者。

## 7. 验收用例（最少覆盖）
- 用例1：订单下发后无需设计，能从已下发直接转备货中。
- 用例2：订单下发后需要设计，必须先到设计中，再到备货中。
- 用例3：备货中缺料可进入采购流程，主状态仍保持备货中。
- 用例4：备货中完成后可转生产中，且展示区域变化正确。
- 用例5：生产完成后仅厂长可转待出货。
- 用例6：待出货时若订单部未放行，厂长不可最终出货确认。

## 8. 部门架构与权限矩阵（建议落地版）

### 8.1 设计原则
- 页面与文案只展示“部门”，不再绑定具体职称。
- 系统实际控制按“动作权限”执行，不按部门名称硬编码。
- 同一部门可多人，是否可操作由权限键决定。

### 8.2 部门定义（建议）
- 订单部：负责建单、下发、出货放行、结案审批。
- 生产部：负责生产状态推进（已下发/设计中/备货中/生产中）。
- 采购部：负责采购申请与执行。
- 财务部：负责应收与发起结案。
- 管理员：全量权限与维护配置。

### 8.3 动作权限矩阵（按当前系统权限键）
| 权限键 | 含义 | 管理员 | 订单部 | 生产部 | 采购部 | 财务部 |
|---|---|---|---|---|---|---|
| `action_mo_create` | 新建生产订单 | Y | Y | N | N | N |
| `action_mo_issue` | 草稿下发（草稿->已下发） | Y | Y | N | N | N |
| `action_mo_workshop_judge` | 车间判读与推进（已下发/设计中/备货中/生产中） | Y | N | Y | N | N |
| `action_mo_to_ship` | 生产完成转待出货 | Y | N | Y | N | N |
| `action_ship_submit` | 出货确认提交 | Y | N | Y | N | N |
| `action_pr_submit` | 提交采购申请审批 | Y | N | N | Y | N |
| `action_approve_pr` | 审批采购申请 | Y | Y | Y | N | N |
| `action_finance_apply_close` | 申请结案 | Y | Y | N | N | Y |
| `action_approve_close` | 结案审批同意/退回 | Y | Y | N | N | N |

### 8.4 菜单权限矩阵（按当前菜单键）
| 菜单键 | 含义 | 管理员 | 订单部 | 生产部 | 采购部 | 财务部 |
|---|---|---|---|---|---|---|
| `dashboard` | 工作台 | Y | Y | Y | Y | Y |
| `production_order` | 生产订单 | Y | Y | Y | Y(只读为主) | Y(只读为主) |
| `purchase` | 采购 | Y | N | Y(查看对料) | Y | N |
| `shipping` | 出货 | Y | Y(查看) | Y | N | N |
| `finance` | 财务应收 | Y | Y | N | N | Y |
| `approval` | 审批中心 | Y | Y | Y(采购审批) | N | Y(查看/提交) |
| `roles` | 角色管理 | Y | N | N | N | N |

### 8.5 关键约束（必须保持）
- “订单部就是厂长”的现实组织关系可以存在，但系统中建议视为两个部门职责；若确需合并，可给同一账号同时授予两组权限。
- 状态按钮显示必须由权限键控制，不由部门名称字符串控制。
- 草稿可见性仅订单部与管理员，避免未下发订单提前扩散。
